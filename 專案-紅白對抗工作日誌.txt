### [2026-03-13 修復漏洞紀錄]

#### 🛠️ 今日修復目標：多人併發投票異常處理
根據上週實戰測試回饋，當多人同時湧入投票時，系統會出現以下嚴重問題：
1. **網頁自動刷新**：投票動作觸發了不必要的頁面重導或刷新。
2. **票數歸零**：高併發下，伺服器端的變數處理可能產生競爭條件 (Race Condition)，導致數據遺失。
3. **匯入檔案狀態消失**：系統狀態管理不穩定，導致本應存在的匯入數據在併發過程中遺失。

#### 🔍 預計處理方向：
- 檢查 `main.py` 中的全局變數使用情況，評估是否需要鎖定機制或改用資料庫存儲。
- 優化前端 `index.html` 的 AJAX 請求，防止表單預設提交導致的刷新。
- 檢查 Vercel 無伺服器環境 (Serverless) 的狀態持久化限制。

#### ✅ 今日已完成修復：
1. **實作 Append-only 投票機制**：
   - 新增 `VotesLog` 分頁。投票時改為 `append_row` 紀錄，徹底杜絕多人在同一儲存格競爭導致票數歸零。
   - 後端 `/status` 與 `/end_round` 會自動彙整該回合的所有得票數。
2. **強化 Serverless 狀態持久化**：
   - 在所有關鍵進入點（`/status`, `/admin`, `/vote` 等）加入 `load_data()` 強制檢查。
   - 當 Vercel 實例重啟導致記憶體清空時，系統能立即從 Google Sheets 恢復作品清單，解決「匯入狀態消失」問題。
3. **前端穩定性優化**：
   - 加入投票按鈕鎖定 (isVoting 旗標)，防止重複點擊。
   - 加入投票中 Loading 提示。
   - 優化狀態輪詢 (checkStatus) 的錯誤容忍度，避免網路波動導致 UI 異常。
4. **版本更新至 v1.1.9**：
   - 更新 `UPDATE_LOG` 與 `APP_VERSION` 以識別修復版本。

### [2026-03-18 壓力測試與環境優化紀錄]

#### 🛠️ 今日目標：實測 30 人併發投票並建立本地開發環境 SOP
1. **併發壓力測試**：使用 `load_test.py` 模擬 30 人同時投票。
2. **環境變數配置**：協助使用者建立 `.env` 檔案，整合 Google Sheets 與 Cloudinary。
3. **穩定性強化**：解決併發時 Google Sheets API 的 Quota Exceeded (429) 錯誤。

#### ✅ 今日已完成修復：
1. **實作 GSheet 寫入重試機制**：
   - 在 `gsheet_manager.py` 的 `record_vote` 中加入了 **Exponential Backoff (指數退避)** 與 **Retry (重試)** 機制。
   - 當 Google API 回報 429 (配額上限) 或 503 (服務忙碌) 時，系統會自動排隊重試，最高重試 10 次。
   - 經測試，30 人併發時即使出現 API 忙碌，最終也能 100% 成功寫入。
2. **建立本地開發環境 (.env)**：
   - 引導使用者配置 `GOOGLE_SHEET_ID` 與 `GOOGLE_CREDENTIALS`。
   - 微調 `main.py` 加入 `load_dotenv()`，使本地開發環境與 Vercel 雲端環境完全一致。
3. **撰寫對外通知與 SOP**：
   - 建立 `雲端資料庫與圖床設定指南.txt`，詳細記錄 Google Sheets 與 Cloudinary 的串接流程。
4. **自動化數據同步**：
   - 優化 `load_data` 邏輯，當 Google Sheets 為空時自動從本地 `works.json` 備份數據並同步回雲端。

#### 📊 壓力測試結果：
- **測試規模**：30 人併發投票 (concurrent POST)
- **成功率**：100% (透過重試機制克服了 API 限流)
- **最終數據狀態**：Google Sheets `VotesLog` 完整收到 30 筆測試紀錄。
